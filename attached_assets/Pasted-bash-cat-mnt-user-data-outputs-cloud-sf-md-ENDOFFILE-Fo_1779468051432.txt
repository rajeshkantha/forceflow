## 1. Vision & North Star {#vision}

Salesforce teams today face three brutal bottlenecks:

- **Dev time** — writing Apex, configuring metadata, and setting up validations manually takes days per feature.
- **Deployment time** — manual change sets, environment drift, and approval chains stretch releases to weeks.
- **Testing time** — writing test classes, running regression manually, and chasing 75% code coverage is a recurring tax.

ForceFlow eliminates all three. It is a **company-scoped, multi-tenant platform** where each company (tenant) brings their own Salesforce orgs, their own team, and their own model configuration. A Frontier-role user (the company's super-admin) onboards the company, connects orgs, and invites role-based agent users. Every AI agent operates with full awareness of that company's Salesforce implementation — not generic Salesforce knowledge.

**Target outcome:** A feature that takes a Salesforce developer 3 days of dev + 2 days of deployment + 1 day of QA completes in under 4 hours on ForceFlow.

**Every development request goes through Plan → Approve → Execute.** No agent ever mutates a Salesforce org without a human-reviewed plan.

## 2. Multi-Tenancy Architecture {#multi-tenancy}

### Tenant = Company

ForceFlow is a **company-scoped SaaS platform**. Every resource — orgs, users, model configs, chat threads, pipelines, agent memory — belongs to a `tenant` (company). Users belong to one tenant. Data is never shared across tenants.

### Isolation Model

```
┌─────────────────────────────────────────────────────────────────┐
│  TENANT: Acme Corp (tenant_id = abc123)                         │
│                                                                  │
│  Users: alice@acme.com (Frontier), bob@acme.com (Developer),   │
│         carol@acme.com (Sales), dave@acme.com (Support)         │
│                                                                  │
│  Orgs: Acme Production, Acme UAT, Acme Dev Sandbox             │
│  Model Config: Anthropic claude-sonnet-4 (tenant-level BYOK)   │
│  Pipelines, chat threads, QA plans — all scoped to tenant       │
└─────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────┐
│  TENANT: BetaCo (tenant_id = def456)                            │
│  — completely isolated, no data overlap —                       │
└─────────────────────────────────────────────────────────────────┘
```

### RLS Policy

Every database table carries `tenant_id`. Row-Level Security enforces:

```sql
-- Example policy on salesforce_orgs
CREATE POLICY tenant_isolation ON salesforce_orgs
  USING (tenant_id = current_setting('app.current_tenant_id')::uuid);
```

The `current_tenant_id` is set per request from the verified JWT claim. No query can ever return rows from another tenant.

### Subdomain Routing (Optional, V6)

Tenants can optionally get `acme.forceflow.app` — a subdomain that auto-scopes the session to their tenant. Custom domain support available in Enterprise tier.

## 3. Company Onboarding Flow {#onboarding}

### Overview

When a new company signs up, the first user (who becomes the **Frontier** / super-admin) goes through a structured onboarding wizard before reaching the product. This collects the company context that all AI agents will use throughout the lifetime of the tenant.

### Onboarding Wizard — Step by Step

```
Step 1: Account Creation
  ├── Email + password
  ├── Full name
  └── Accept terms of service

Step 2: Company Profile
  ├── Company name (e.g. "Acme Corp")
  ├── Company website (optional, used for org labeling)
  ├── Industry vertical
  │     [Manufacturing] [Financial Services] [Healthcare]
  │     [Retail / CPG]  [Technology]         [Non-Profit]
  │     [Professional Services]              [Other]
  ├── Company size
  │     [1–50 employees] [51–500] [501–5000] [5000+]
  ├── Salesforce Clouds in use (multi-select)
  │     [Sales Cloud] [Service Cloud] [Experience Cloud]
  │     [CPQ / Revenue Cloud] [Marketing Cloud] [Field Service]
  │     [Health Cloud] [Financial Services Cloud] [Custom / Other]
  └── Brief description of what your Salesforce org does
        (free text, 2–5 sentences — this seeds agent system prompts)

Step 3: Salesforce Org Setup
  ├── "Connect your first Salesforce org"
  ├── Org label (e.g. "Production", "Main Org")
  ├── Org type: [Production] [Sandbox] [Developer Edition] [Scratch Org]
  ├── Login URL: [login.salesforce.com] [test.salesforce.com] [Custom domain]
  ├── Connected App Client ID + Client Secret
  │     (with inline guide: "How to create a Connected App →")
  └── OAuth redirect + Connect button → SF OAuth flow

Step 4: Model Configuration
  ├── "Choose how your AI agents are powered"
  ├── Provider: [OpenAI] [Anthropic] [OpenRouter] [Custom Endpoint]
  ├── API Key (encrypted at rest)
  ├── Model name (with recommended defaults per provider)
  └── Test connection → shows "Model ready ✓"

Step 5: Invite Your Team (optional, skip-able)
  ├── Enter email addresses
  ├── Assign initial role per invite:
  │     [Frontier] [Developer] [Sales] [Support] [SME]
  └── Send invites → invited users land on Step 1 with tenant pre-filled

Step 6: Done → Dashboard
  └── "Welcome to ForceFlow, Acme Corp. Your agents are ready."
```

### What Onboarding Data Powers

The company profile collected in Step 2 is stored in `tenant_context` and injected into every agent's system prompt as a preamble:

```
You are a Salesforce AI agent working for Acme Corp, a manufacturing 
company with 500–5000 employees. They use Sales Cloud, Service Cloud, 
and CPQ. Their Salesforce org manages B2B sales pipeline, field service 
work orders, and customer support cases for industrial equipment. 
Always ground your analysis and recommendations in this business context.
```

This makes every agent — Sales, Support, SME, Frontier — contextually aware of the company without the user having to re-explain it in every conversation.

## 4. V1 — Foundation {#v1-foundation}

### What Ships

**Account & Workspace**
- Email/password signup + signin (Lovable Cloud auth).
- Tenant created on first signup; all subsequent users join via invite.
- All resources RLS-scoped to `tenant_id`.

**Salesforce Org Connections (Multi-Org per Tenant)**
- Add orgs via Salesforce OAuth 2.0 Web Server flow with PKCE.
- Each org belongs to the tenant, visible to all tenant members (per their role).
- Tokens stored AES-256-GCM encrypted, auto-refreshed on expiry.
- Org list page: status indicator, instance URL, username, last sync, org type badge, delete/reconnect.

**Model Configuration (Tenant-Level BYOK)**
- Add one or more model providers: OpenAI, Anthropic, or custom OpenAI-compatible endpoint.
- Model configs belong to the tenant; Frontier manages them, all agents use them.
- Keys stored encrypted server-side; never exposed to the client.

**Configuration Agent (Chat)**
- Chat UI with thread list, streaming responses, and tool-call rendering with before/after diffs.
- Org + model selectors per thread.
- Agent tools (Salesforce REST / Tooling / Metadata API):
  - `sf_soql_query` — read data and metadata
  - `sf_describe_object` — inspect object schema
  - `sf_list_metadata` — list components by type
  - `sf_read_metadata` — read metadata XML
  - `sf_create_custom_field` — add fields to objects
  - `sf_create_validation_rule` — create validation rules
  - `sf_assign_permission_set` — assign permission sets to users
  - `sf_deploy_metadata` — generic metadata deploy (confirmation gate)
- All mutating tools require explicit user approval before execution.

**Dashboard**
- Tenant-scoped: connected orgs count, team members, active model, recent threads, quick-prompt suggestions.

## 5. V2 — Multi-Tenant · Role-Based Agents · Frontier Role · Smart Agent Intelligence {#v2-agents}

### 5.1 Extended Model Providers

- `openrouter` — base URL pinned to `https://openrouter.ai/api/v1`, free-form model slug.
- `aggregator` — user-supplied base URL + model slug (Together AI, Groq, Fireworks, LiteLLM, vLLM, etc.).
- Both reuse `createOpenAICompatible`; OpenRouter adds `HTTP-Referer` / `X-Title` headers.

### 5.2 Role-Based Agents — Full Roster

Each user in a tenant is assigned a role. The role determines which agent they access, what tools are exposed, and what the system prompt says. Roles are **not just UI labels** — they are full agent personas with dedicated tool surfaces, refusal patterns, and cross-role handoff instructions.

#### Role: Frontier (Super-Admin / Power User)

The Frontier role replaces "Developer" as the highest-privilege agent. The Frontier user is the company's Salesforce owner — typically an architect, lead admin, or CTO. They can do everything any other role can do, plus:

- **Onboard and manage all team members** — invite users, assign roles, revoke access.
- **Manage all org connections** — add, reconnect, delete orgs.
- **Manage model configuration** — set tenant-wide AI provider and keys.
- **Access all agent roles** — can switch to Sales, Support, SME, or Developer agent context in any thread.
- **Approve production deployments** — only Frontier can approve the final pipeline gate to Production.
- **View audit log** — full record of every action every team member has taken.
- **Org governance** — org health dashboard, drift detection, dead code reports.

The Frontier agent system prompt:

```
You are the Frontier Agent for {company_name}. You have full access to 
all Salesforce orgs, all configuration tools, and all team management 
functions. You are the most senior AI agent on this platform. 
You think in terms of architecture, long-term maintainability, and 
business outcomes. Before executing any mutating action, you generate 
a detailed plan and require explicit approval. You also oversee the 
team — you can onboard agents, assign roles, and see everything.
```

**Frontier-exclusive tools:**
- `tenant_invite_user` — send invite to a new team member with a specified role
- `tenant_list_members` — list all team members + roles + last active
- `tenant_update_member_role` — change a user's role
- `tenant_revoke_member` — remove a user from the tenant
- `tenant_org_health_report` — generate cross-org health summary
- `tenant_audit_log` — query audit log for any actor/action
- All tools from Developer + SME + Sales + Support roles

#### Role: Developer

Configuration and code changes. The technical implementor.

**Tools:** `sf_soql_query`, `sf_describe_object`, `sf_list_metadata`, `sf_read_metadata`, `sf_explain_validation_rules`, `sf_explain_flow`, `sf_sharing_check`, `sf_create_custom_field`, `sf_create_validation_rule`, `sf_assign_permission_set`, `sf_deploy_metadata`, `sf_apex_execute`, `sf_run_anonymous`, `sf_create_apex_class`, `sf_create_lwc`, `sf_create_flow`, `sf_create_trigger`

**Refusal pattern:** Declines user management and org connection tasks; directs to Frontier.

#### Role: Sales Agent

Pipeline, opportunities, quotes, leads, follow-ups, daily data work.

**Tools:** `sf_soql_query`, `sf_create_record`, `sf_update_record`, `sf_create_opportunity`, `sf_create_quote`, `sf_create_task`, `sf_log_call`, `sf_send_email_template`, `sf_get_my_pipeline`, `sf_lead_followup`, `sf_dedupe_merge`, `sf_forecast_summary`

**Refusal pattern:** Declines all metadata/config changes; suggests Developer or Frontier agent.

#### Role: Support Agent

Cases, knowledge articles, SLAs, escalations, customer satisfaction.

**Tools:** `sf_soql_query`, `sf_create_case`, `sf_update_case`, `sf_post_case_comment`, `sf_search_knowledge`, `sf_escalate_case`, `sf_sla_check`, `sf_create_entitlement`, `sf_close_case`

**Refusal pattern:** Declines schema changes; suggests SME for "why" questions, Developer for fixes.

#### Role: SME Agent (Subject Matter Expert)

Read-only deep expertise. The "explain everything" agent.

**Tools (all read-only):** `sf_soql_query`, `sf_describe_object`, `sf_list_metadata`, `sf_read_metadata`, `sf_explain_validation_rules`, `sf_explain_flow`, `sf_sharing_check`, `sf_tooling_query`, `sf_dependency_query`, `sf_explain_trigger`, `sf_explain_apex_class`

**Refusal pattern:** Never mutates. Declines all write operations; suggests Developer for changes.

## 6. V3 — Git Pipeline & SDLC Engine {#v3-pipeline}

### Overview

The core SDLC acceleration layer. Git-backed environment promotion pipeline (Dev → QA → UAT → Production) with automated deployments triggered by branch events, and deep Jira/Linear ticket integration.

### Git Integration (GitHub / GitLab / Bitbucket)

- Connect a repository per Salesforce project (Frontier manages connections).
- Branch strategy enforced by ForceFlow:
  - `feature/*` → Developer sandbox
  - `qa` → QA sandbox
  - `uat` → UAT sandbox
  - `main` → Production
- Pull request events trigger automated pipeline runs.
- Salesforce metadata stored in SFDX project format (source-tracked, version-controlled).

### Pipeline Stages

```
[Feature Branch] → PR Opens
        ↓
[Stage 1] Static Analysis (PMD + ESLint + Flow Scanner) — blocks on CRITICAL violations
        ↓
[Stage 2] Deploy to Developer Sandbox — validate-only first, then deploy
        ↓
[Stage 3] Run Apex Tests — 75% coverage gate + all specified tests pass
        ↓
[Stage 4] PR Merge → qa branch
        ↓
[Stage 5] Auto-deploy to QA Sandbox
        ↓
[Stage 6] QA Agent Smoke Tests (V5)
        ↓
[Stage 7] Promote to UAT — Frontier approval gate (1-click)
        ↓
[Stage 8] UAT sign-off → merge to main
        ↓
[Stage 9] Production Deploy — Frontier approval required — with rollback plan
```

### Deployment Strategies

- **Validate-Only First:** Check-only deploy before real deploy; halts with diff on failure.
- **Delta Deployments:** `sfdx-git-delta` computes only changed components — cuts deploy time from 15–20 min to under 2 min for small changesets.
- **Rollback:** Pre-deployment snapshot stored; one-click redeploys prior known-good state.
- **Unlocked Packages (optional):** Package version promotion for DX-native projects.

### Ticket Integration (Jira / Linear)

- Link runs to tickets via commit convention: `[PROJ-123]` or `closes #42`.
- Auto-update ticket status as pipeline progresses.
- Attach deployment logs + test results as ticket comments.
- Release notes auto-generated from ticket titles in a merge window.

## 7. V4 — Code Quality & Static Analysis {#v4-quality}

### Analysis Engines

| Tool | Scope | Integration |
|---|---|---|
| **PMD (Apex)** | SOQL in loops, DML in loops, governor limit risks, security violations | CI pipeline; violations annotated on PR diff |
| **ESLint (LWC/Aura)** | JS quality, accessibility, security patterns | CI pipeline |
| **Flow Scanner** | Hardcoded IDs, missing null checks, bulkification in Flows | CI pipeline |
| **Salesforce Code Analyzer v5** | Unified runner (440+ rules, PMD + ESLint + RetireJS); AI quick-fix via ApexGuru | Available April 2025 |
| **Security Scan** | Hardcoded credentials, SOQL injection, XSS in LWC | Part of Code Analyzer v5 |

### Violation Severity Gates

```
CRITICAL (blocks merge):
  - SOQL/DML inside loops
  - Hardcoded IDs or credentials
  - Empty catch blocks
  - Missing sharing keyword on Apex classes

HIGH (blocks merge, suppressible with justification):
  - Methods > 200 lines
  - Cyclomatic complexity > 20
  - No bulk handling in triggers

MEDIUM (warns, does not block):
  - Unused variables, missing null checks

LOW (informational):
  - Style inconsistencies, missing comments
```

### AI Quick-Fix

For CRITICAL and HIGH violations, ForceFlow offers "AI Fix" — the Developer/Frontier agent generates a corrected version for review. One-click apply commits the fix to the branch.

### Coverage Enforcement

- Minimum 75% Apex code coverage (platform minimum); ForceFlow default is 85% (configurable).
- Test gap analysis surfaces which classes/methods have no coverage.
- AI Test Generation: Developer/Frontier agent generates Apex test class skeletons on demand.

## 8. V5 — Autonomous QA Testing Agent {#v5-qa}

### How It Works

1. After each deployment to QA sandbox, the QA Agent receives the deployed component manifest.
2. It generates a test plan from changed components (new field → render check; new validation rule → invalid-input test).
3. It executes the test plan using Playwright (UI) and REST API (data assertions).
4. Produces a structured report: passed, failed, screenshots, suggested fixes.
5. Results posted to pipeline run and linked Jira/Linear ticket.

### Test Types

| Test Type | Mechanism |
|---|---|
| Field Existence Check | SOQL query + Metadata API page layout inspection |
| Validation Rule Test | REST API record create with invalid data; assert error message |
| Flow Execution Test | Data insert triggers flow; check output records via SOQL |
| Permission Check | Log in as profile/PS user; assert field/object visibility |
| Page Layout Render | Playwright screenshot + computer vision field presence check |
| Apex Test Re-run | Execute changed class tests; surface failures with stack traces |
| Integration Smoke Test | Call registered external endpoints; assert expected responses |

)

  id uuid, tenant_id uuid, test_plan_id uuid, pipeline_run_id uuid,
  status text, results jsonb, duration_ms integer, created_at timestamptz

## 9. V6 — Analytics, Governance & Enterprise {#v6-enterprise}

## 10. Plan & Execute Mode — How All Development Requests Work {#plan-execute}

### The Core Principle

**No agent in ForceFlow ever directly executes a mutating action on a Salesforce org without a human-reviewed plan.** This applies to every role, every org, every change — no exceptions.

The flow for every development or configuration request:

```
User Request
     ↓
[ANALYSIS PHASE]
  Agent interrogates the org:
  - What currently exists related to this request?
  - What is the current implementation coverage?
  - What components are involved?
  - What are the dependencies?
     ↓
[PLAN GENERATION]
  Agent produces a structured execution plan:
  - Step-by-step actions in order
  - Each step: what, why, estimated effort, risk level
  - Total estimated effort
  - Rollback strategy
  - Open questions / assumptions
     ↓
[HUMAN REVIEW]
  Plan rendered in UI as an interactive card:
  - User can edit, reject, or approve each step
  - User can ask the agent to revise the plan
  - "Approve & Execute" button
     ↓
[EXECUTION PHASE]
  Agent executes steps in sequence:
  - Each step shows real-time status (pending / running / done / failed)
  - Each tool call shows before/after diff
  - Failures stop execution; agent explains and offers remediation plan
     ↓
[VERIFICATION]
  Agent verifies the outcome:
  - Re-queries the org to confirm changes applied correctly
  - Runs relevant Apex tests if code was deployed
  - Produces a deployment summary report
```

### Plan Card UI

Every plan generated by the agent is rendered as a structured card in the chat:

```
┌──────────────────────────────────────────────────────────────────┐
│  📋 PLAN: Add "Renewal Date" field to Opportunity               │
│  Org: Acme Production  |  Est. effort: ~12 min  |  Risk: LOW    │
├──────────────────────────────────────────────────────────────────┤
│  CURRENT STATE ANALYSIS                                          │

├──────────────────────────────────────────────────────────────────┤

│     → Assign to "Standard" field-level security                  │
│  2. Add field to "Opportunity Layout" page layout                │
│  3. Add field to "Sales" Permission Set field permissions        │
│  4. Verify field visible on record page (QA check)              │
├──────────────────────────────────────────────────────────────────┤

### Modes by Complexity

| Request Type | Analysis Depth | Plan Size | Typical Time |
|---|---|---|---|
| Simple (add a field, update a rule) | Shallow org scan (1–2 tool calls) | 2–5 steps | < 5 min |
| Medium (new Flow, Apex trigger) | Component analysis + dependency scan | 5–10 steps | 10–20 min |
| Complex (multi-object feature, integration) | Deep org analysis + impact assessment | 10–25 steps | 30–90 min |
| Enterprise (architectural change) | Full org audit + cross-org impact | 25+ steps, phased | Async / multi-session |

### Re-Planning

If execution fails at any step, the agent does not silently stop. It:
1. Logs the failure with full error details.
2. Generates a remediation plan for the failed step.
3. Presents it for user approval before re-attempting.
4. Rolls back successfully executed steps if the failure is unrecoverable.

## 11. Smart Agent Intelligence Layer {#agent-intelligence}

### Overview

Every agent in ForceFlow is trained (via system prompt + tool access) to function as a domain expert who knows the specific company's Salesforce org. When a user describes a business requirement, the agent performs a **three-phase intelligence cycle** before generating any plan.

### Phase 1: Business Requirement Analysis

The agent parses the request and structures it:

```
User: "We need to track when a customer last renewed their subscription 
       and automatically flag accounts that haven't renewed in 12 months."

Agent Analysis:
  Business Objective: Renewal tracking + at-risk account identification
  Affected Objects: Account, Opportunity (likely), Contract (possible)
  Process Type: Data capture + automated alerting
  Complexity Estimate: Medium
  Cross-cloud concern: Sales Cloud + possibly Service Cloud
```

### Phase 2: Current Implementation Assessment

Before suggesting anything, the agent deeply queries the org to understand what already exists:

**Tools invoked (read-only, in sequence):**
1. `sf_soql_query` — search for existing renewal-related fields on Account, Opportunity, Contract
2. `sf_list_metadata` — list existing Flows for "renewal" or "churn" patterns
3. `sf_describe_object` — inspect Account custom fields for date/status tracking fields
4. `sf_tooling_query` — check for existing Apex triggers or scheduled jobs related to renewal logic
5. `sf_list_metadata (CustomNotificationTypes)` — check existing notification setup

**Output — Current State Report card:**

```
┌──────────────────────────────────────────────────────────────────┐
│  🔍 CURRENT IMPLEMENTATION ANALYSIS                              │
│  Requirement: Renewal tracking + at-risk flagging               │
├──────────────────────────────────────────────────────────────────┤
│  WHAT EXISTS (Partial Implementation — ~30% covered)            │
│                                                                  │
│  Fields found:                                                   │
│  ✓ Account.Contract_End_Date__c (Date) — exists, populated on   │
│    47% of Account records                                        │
│  ✓ Opportunity.CloseDate — used as a proxy for renewal date in   │
│    some records (not reliable)                                   │
│  ✗ No dedicated "Last_Renewal_Date__c" field found              │
│  ✗ No "Renewal_Status__c" or at-risk flag field found           │
│                                                                  │
│  Automations found:                                              │

│    before Contract_End_Date__c. Active. (partial match)         │

- Builds **on top of** what already exists (reuses `Contract_End_Date__c`, extends the existing Flow rather than creating a duplicate).
- Sequences steps in safe deployment order (fields before Flows before reports).
- Includes effort estimates per step based on component type.
- Flags dependencies and risks.

### Continuous Intelligence — Agent Memory per Org

Each org accumulates an **Agent Knowledge Base** — a structured, queryable store of:

- Discovered metadata (fields, objects, Flows, Apex classes) indexed by function/purpose.
- Past analyses and implementation plans (so the agent doesn't re-scan what it already knows).
- Implementation decisions and rationale (searchable: "why was this field created this way?").
- Dependency graph (updated after every deployment).

This means the second time a user asks about a topic the agent has already analyzed, it answers from its knowledge base instantly — not from a full re-scan.

## 14. SDLC Time Reduction Strategy {#sdlc-strategy}

### Current State vs ForceFlow

| SDLC Phase | Manual Today | ForceFlow Target | Reduction |
|---|---|---|---|
| Requirement → Design | 4–8 hrs (discovery, design doc) | 30 min (SME/Frontier Agent analyzes org, delivers current-state + gap report) | ~85% |
| Dev: writing Apex/config | 1–3 days | 2–4 hrs (plan generated, Developer Agent executes approved steps) | ~80% |
| Code review & quality | 4–8 hrs (manual PR review) | 15 min (automated PMD/ESLint gate + AI quick-fix) | ~95% |
| Deploy to Dev sandbox | 30–60 min (change set) | 2 min (delta deploy via SFDX on PR open) | ~97% |
| Write Apex test classes | 4–8 hrs | 30 min (AI generates skeletons; developer fills assertions) | ~90% |
| Deploy to QA | 30–60 min | Automatic (triggered by merge to `qa`) | ~100% |
| QA smoke testing | 1–2 days (manual) | 20–40 min (QA Agent + visual regression) | ~95% |
| Deploy to UAT | 30–60 min | Automatic + Frontier 1-click gate | ~90% |
| UAT sign-off | 1–3 days | 2–4 hrs (QA Agent pre-validates; human spot-checks) | ~80% |
| Production deploy | 2–4 hrs | 10 min (pipeline + rollback-ready; Frontier approves) | ~95% |
| **Total (typical feature)** | **~5–10 days** | **~6–10 hours** | **~85–90%** |

### Principles

**Plan Before Execute** — every action is pre-planned, reviewed, and approved. Humans make decisions; agents do work.

**Context-Aware Intelligence** — agents know the company's industry, Salesforce clouds, and existing implementation. Analysis is specific, not generic.

**Shift Left on Quality** — static analysis and test generation happen before code leaves the developer. Issues are cheapest to fix when the developer is still in context.

**Delta Deployments** — `sfdx-git-delta` deploys only changed components. Most feature deploys complete in under 2 minutes.

**Org Knowledge Accumulates** — the knowledge base grows with every analysis. The second time someone asks about a topic the agent has already scanned, it answers instantly.

**Approval-Gated Automation** — humans approve, machines execute. The agent assembles payloads and diffs; the human reviews and approves with one click.

## 15. Technology Stack {#tech-stack}

### Frontend
- **Framework:** TanStack Start (file-based routing, SSR, server functions)
- **UI Components:** shadcn/ui + Radix primitives
- **Styling:** Tailwind CSS — dark slate/indigo design system
- **Fonts:** Inter (UI), JetBrains Mono (code/IDs)
- **Streaming:** AI SDK `useChat` hook with tool-call rendering + plan card components
- **State:** TanStack Query for server state

### Backend
- **Runtime:** Lovable Cloud server functions + route handlers
- **AI:** Vercel AI SDK `streamText`, provider-agnostic adapter; role system prompt + tenant preamble injected per request
- **SF Client:** Axios-based, token-aware, auto-refresh, retry on 401; composite API for batch operations
- **Pipeline Runner:** Node.js child processes for SFDX CLI commands
- **QA Automation:** Playwright (headless Chromium) in worker pool
- **Static Analysis:** PMD CLI, ESLint, Salesforce Code Analyzer v5 — subprocesses in pipeline stages

### Database
- **Primary:** Supabase (Postgres) via Lovable Cloud — all tables tenant-scoped with RLS
- **Encryption:** AES-256-GCM for all tokens and API keys at rest; decrypted only in server function scope
- **Realtime:** Supabase realtime for pipeline run status + plan execution step updates

### Integrations
- **Salesforce:** REST API, Tooling API, Metadata API (SOAP + REST), Composite API
- **Git:** GitHub Apps (webhooks + API), GitLab webhooks, Bitbucket webhooks
- **Tickets:** Jira REST API v3, Linear GraphQL API
- **Notifications:** Slack Incoming Webhooks, Microsoft Teams webhooks
- **MCP:** Model Context Protocol server (SSE transport) wrapping all `sf_*` tools

### Infrastructure
- **CI Quality Tools:** Salesforce Code Analyzer v5 (PMD + ESLint + RetireJS, 440+ rules), `sfdx-git-delta`
- **Package Registry:** Salesforce Unlocked Packages (optional)
- **Secrets:** AES-256-GCM encryption; keys never logged or exposed to client
